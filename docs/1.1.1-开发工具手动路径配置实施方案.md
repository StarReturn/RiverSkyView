# 江天一览 RiverSkyView 1.1.1 开发工具手动路径配置实施方案

> 文档版本：1.0  
> 目标版本：1.1.1  
> 编制日期：2026-07-15  
> 适用平台：Windows 10/11 x64  
> 实施性质：第二期编辑器/IDE 启动能力可靠性补强

## 一、实施结论

1.1.1 将把“自动检测”调整为辅助能力，并允许用户为每个内置编辑器或 IDE 手动指定可执行文件。

路径使用优先级固定为：

```text
有效的用户手动路径
  > 有效的自动检测路径
  > 未配置
```

手动配置失效时，系统必须明确提示用户修复，不得静默切换到其他安装版本。只有用户主动执行“恢复自动检测”后，系统才重新使用自动检测路径。

本次实施不修改项目文件夹名称，不修改现有应用数据目录、数据库文件名、bundle identifier 和项目标记文件名称。

## 二、问题背景

当前 1.1.0 已实现：

- VS Code、Cursor、Visual Studio、JetBrains 系列和 Android Studio 自动检测；
- 自定义编辑器 `.exe` / `.com` 配置；
- 全局默认编辑器和项目级默认编辑器；
- 项目目标发现、选择及记忆；
- 使用参数数组安全启动开发工具。

当前缺口是：内置开发工具的检测路径只能查看，不能直接修改。用户虽然可以另建“自定义编辑器”，但这会产生重复项，且不能直接纠正 `builtin:vscode`、`builtin:cursor` 等内置项的路径。

容易出现的实际问题包括：

- 软件安装在非默认磁盘或便携目录；
- 同一工具同时安装稳定版、预览版或多个版本；
- JetBrains Toolbox 路径结构发生变化；
- 系统 PATH 指向旧版本；
- 自动检测找到卸载残留；
- 用户希望固定使用某个特定版本。

## 三、目标与非目标

### 3.1 实施目标

1. 每个内置开发工具都可以手动选择 `.exe` 或 `.com`。
2. 用户可以查看当前生效路径及其来源。
3. 用户可以测试启动、重新检测、恢复自动检测和打开程序所在目录。
4. 手动路径优先于自动检测，刷新检测不得覆盖手动配置。
5. 项目列表、项目详情、全局默认和项目默认统一使用同一套路径解析逻辑。
6. 路径失效时提供可理解、可恢复的错误提示。
7. 保持 1.1.0 数据和编辑器偏好无损升级。

### 3.2 非目标

- 不扫描所有磁盘寻找任意可执行文件。
- 不支持 `.bat`、`.cmd`、`.ps1`、`.lnk` 或 Shell 命令。
- 不自动下载或安装开发工具。
- 不修改开发工具自身配置。
- 不把编辑器路径写入项目目录。
- 不要求识别所有第三方 IDE；未知工具继续通过“自定义编辑器”添加。

## 四、产品与交互设计

### 4.1 设置入口

保留现有入口：

```text
设置 → 编辑器与 IDE
```

将现有“检测结果表格”升级为“开发工具路径管理表格”。

| 字段 | 内容 |
|---|---|
| 开发工具 | 名称及内置图标 |
| 当前路径 | 实际启动时使用的绝对路径 |
| 路径来源 | 手动指定 / 自动检测 / 未配置 |
| 状态 | 可用 / 路径失效 / 未配置 / 验证失败 |
| 操作 | 手动选择、重新检测、测试、恢复自动、打开位置 |

状态展示规则：

- `可用`：绿色；
- `路径失效`：红色；
- `未配置`：灰色；
- `验证失败`：橙色，并显示具体原因。

### 4.2 手动选择流程

1. 用户点击某个内置工具的“手动选择”。
2. 打开 Windows 原生文件选择器。
3. 文件选择器只显示 `.exe` 和 `.com`。
4. 用户选择文件后，后端立即校验。
5. 校验成功后显示文件路径、识别名称、版本和架构信息（能够读取时）。
6. 用户可以执行“测试启动”。
7. 用户确认保存后，该路径成为当前生效路径。

不强制文件名必须与官方默认名称完全相同，以兼容便携版和重命名文件，但界面应在文件名明显不匹配时显示非阻塞警告。

### 4.3 操作定义

#### 手动选择

选择并保存用户指定的可执行文件。成功后路径来源显示为“手动指定”。

#### 重新检测

仅刷新 `detected_executable`，不得修改或清空 `manual_executable`。

#### 恢复自动检测

清除当前工具的手动路径，并立即重新检测。此操作需要二次确认：

```text
恢复后将不再固定使用当前手动路径，是否继续？
```

#### 测试启动

- VS Code、Cursor、JetBrains 系列：打开应用临时目录下的空测试目录；
- Visual Studio：启动 IDE，不加载用户解决方案；
- 测试只验证系统成功创建进程，不等待 IDE 完全加载；
- 测试目录使用后删除，删除失败则进入应用临时清理队列；
- 测试不得修改任何已登记项目。

#### 打开程序位置

调用 Windows Explorer 定位当前生效的可执行文件。路径失效时该操作禁用。

### 4.4 默认编辑器选择

全局默认编辑器下拉框继续使用稳定的 `editor_key`，不直接保存路径。

规则：

- 当前生效路径有效时可以设为默认；
- 路径失效的工具显示但不可选择；
- 已经成为默认项后路径失效，保留默认关系并显示警告；
- 启动时不得静默换用其他工具；
- 用户可以修复路径或重新选择默认编辑器。

### 4.5 项目列表菜单

项目列表和详情页的编辑器菜单显示：

```text
✓ VS Code              手动指定
✓ Cursor               自动检测
! Visual Studio         路径失效
  PyCharm               未配置
────────────────────────────
  管理开发工具……
```

点击“管理开发工具”跳转到设置页，并定位到编辑器与 IDE 区域。

## 五、数据模型

### 5.1 新增数据库迁移

新增：

```text
src-tauri/migrations/0003_editor_installations.sql
```

建议结构：

```sql
PRAGMA foreign_keys = ON;

CREATE TABLE IF NOT EXISTS editor_installations (
    editor_key TEXT PRIMARY KEY NOT NULL,
    manual_executable TEXT,
    detected_executable TEXT,
    active_source TEXT NOT NULL DEFAULT 'auto'
        CHECK (active_source IN ('auto', 'manual')),
    enabled INTEGER NOT NULL DEFAULT 1,
    verification_status TEXT NOT NULL DEFAULT 'unknown'
        CHECK (verification_status IN ('unknown', 'valid', 'missing', 'invalid', 'launch_failed')),
    detected_source TEXT,
    version TEXT,
    last_detected_at TEXT,
    last_verified_at TEXT,
    last_error TEXT,
    updated_at TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_editor_installations_status
    ON editor_installations(enabled, verification_status);

UPDATE app_meta SET value = '3' WHERE key = 'schema_version';
```

### 5.2 字段说明

| 字段 | 说明 |
|---|---|
| `editor_key` | `builtin:vscode` 等稳定标识 |
| `manual_executable` | 用户明确选择的绝对路径 |
| `detected_executable` | 最近一次自动检测得到的路径 |
| `active_source` | 当前路径来源，`manual` 或 `auto` |
| `enabled` | 是否在项目启动菜单显示 |
| `verification_status` | 最近一次路径或启动校验状态 |
| `detected_source` | PATH、注册表、vswhere、Toolbox、known_path 等 |
| `version` | 能够读取时保存文件版本 |
| `last_error` | 最近一次验证失败的可展示原因 |

### 5.3 生效路径计算

后端必须通过一个统一函数计算：

```rust
fn resolve_active_executable(installation: &EditorInstallation) -> Result<PathBuf>
```

计算规则：

```text
active_source == manual
  → 使用 manual_executable
  → 不存在或无效则返回明确错误
  → 不回退 detected_executable

active_source == auto
  → 使用 detected_executable
  → 不存在或无效则返回未配置/检测失效
```

禁止前端自行拼接或决定最终路径。

### 5.4 兼容策略

- 1.1.0 的 `project_editor_preferences` 保持不变；
- `default_editor_key` 保持不变；
- 现有自定义编辑器 `launch_profiles` 保持不变；
- 首次升级后，为所有内置 `editor_key` 写入或按需生成自动检测记录；
- 数据库迁移不得删除或重写项目默认编辑器关系；
- 降级到 1.1.0 时，新表会被旧版本忽略，但手动内置路径不会生效。

### 6.1 新增文件

```text
src-tauri/src/models/editor_installation.rs
src-tauri/src/repositories/editor_installation_repository.rs
src-tauri/src/services/editor_installation_service.rs
```

也可以合并到现有 `models/editor.rs`，但安装路径解析必须保持独立服务边界。

### 6.2 核心数据结构

```rust
pub struct EditorInstallation {
    pub editor_key: String,
    pub manual_executable: Option<String>,
    pub detected_executable: Option<String>,
    pub active_executable: Option<String>,
    pub active_source: String,
    pub enabled: bool,
    pub verification_status: String,
    pub detected_source: Option<String>,
    pub version: Option<String>,
    pub last_detected_at: Option<String>,
    pub last_verified_at: Option<String>,
    pub last_error: Option<String>,
}
```

`active_executable` 是服务层计算字段，不在数据库重复存储。

### 6.3 Tauri 命令

新增命令：

```rust
list_editor_installations()
set_editor_manual_executable(editor_key, executable)
clear_editor_manual_executable(editor_key)
verify_editor_executable(editor_key, executable?)
test_launch_editor(editor_key)
refresh_editor_detection(editor_key?)
set_editor_enabled(editor_key, enabled)
open_editor_location(editor_key)
```

调整现有命令：

- `list_editors`：从安装路径服务读取当前生效路径和状态；
- `launch_project_editor`：不再直接重新执行自动检测，统一调用 `resolve_active_executable`；
- `set_editor_settings`：验证默认项存在且当前可用；
- `refresh_editor_detection`：只更新检测字段，不影响手动字段。

### 6.4 自动检测结果落库

自动检测服务返回：

```rust
pub struct DetectionResult {
    pub editor_key: String,
    pub executable: Option<PathBuf>,
    pub source: Option<String>,
    pub version: Option<String>,
}
```

刷新流程：

1. 执行原有检测逻辑；
2. 更新 `detected_executable`、`detected_source`、`version` 和时间；
3. 如果当前来源为 `auto`，重新验证检测路径；
4. 如果当前来源为 `manual`，保持手动路径和状态不变；
5. 将两个路径都返回前端，方便用户了解检测差异。

### 6.5 文件验证

保存手动路径前必须验证：

- 是绝对路径；
- 文件真实存在；
- 扩展名为 `.exe` 或 `.com`，大小写不敏感；
- 不是目录、符号链接目标异常或设备路径；
- 路径长度和输入长度处于合理上限；
- 不包含 NUL；
- 能够被当前用户读取；
- 不使用 Shell 展开环境变量。

路径不要求位于 `Program Files`，以支持便携版本。

### 6.6 安全启动

继续使用：

```rust
Command::new(&executable).args(&arguments).spawn()
```

禁止：

- `cmd /c`；
- PowerShell；
- 字符串形式整条命令；
- 用户自定义 Shell 片段；
- 把文件路径和参数拼成一个字符串。

## 八、测试方案

### 8.1 Rust 单元测试

至少覆盖：

1. 手动路径优先于检测路径。
2. 手动路径失效时不回退自动路径。
3. 自动模式正确使用检测路径。
4. 清除手动配置后恢复自动模式。

### 8.3 前端组件测试

## 十一、风险与控制

| 风险 | 控制措施 |
|---|---|
| 检测结果覆盖用户选择 | 手动和检测路径分字段存储，刷新只写检测字段 |
| 手动路径后来失效 | 启动前验证并明确报错，不自动回退 |
| 选择恶意脚本 | 仅允许绝对 `.exe` / `.com`，不使用 Shell |
| 测试启动污染项目 | 使用应用临时空目录，不传真实项目路径 |
| 数据迁移影响原偏好 | 新表增量迁移，使用真实数据库副本复验 |
| UI 操作过于复杂 | 常用操作直接显示，低频操作放入更多菜单 |
| 同工具多个版本选错 | 明确显示当前路径、来源和版本，用户选择为最终权威 |

## 十二、执行约束

交给自动编码工具执行时，必须遵守：

1. 先阅读本方案、第二期实施方案及仓库现状，再修改代码。
2. 不重建项目，不更换现有技术栈。
3. 不修改项目文件夹名称和内部兼容标识。
4. 不删除或覆盖用户已有数据。
5. 不引入 Shell 启动方式。
6. 每完成一个阶段更新实施进度文档。
7. 遇到失败先定位根因，不得以跳过测试作为解决方案。
8. 最终报告必须区分自动化验证和真实 Windows 人工点击验收。
