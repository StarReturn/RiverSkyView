# 界面问题分析报告：热力图、空状态按钮与资料按住显示

> 报告日期：2026-07-15  
> 分析对象：ProjectManage 1.0.0  
> 问题来源：实际安装运行截图与源代码审查  
> 报告性质：问题定位与修复建议，不包含代码修改  
> 综合结论：三个现象均可稳定解释，其中热力图和“按住显示”属于功能缺陷，空状态按钮属于全局 CSS 选择器污染。

## 2. UI-P1-001：活动热力图布局压缩

### 2.1 用户可见现象

从用户截图可见：

- 热力图没有横向铺满卡片。
- 星期标签被挤在左上角，文字纵向重叠。
- 月份标签集中在左上方，无法辨认。
- 色阶图例只占左下角很小范围。
- 热力格子只显示为左下角一小段绿色区域。
- 外层白色卡片宽度正常，说明不是页面整体宽度不足，而是 ECharts 内部画布使用了错误尺寸。

### 2.2 代码路径

项目详情页默认标签是“文件”：

```ts
const activeTab = ref("files");
```

但“活动”标签中的 `ActivityWorkspace` 会随标签结构一起创建：

```vue
<el-tabs v-model="activeTab">
  <el-tab-pane label="文件" name="files">
    <FileWorkspace ... />
  </el-tab-pane>
  <el-tab-pane label="活动" name="activity">
    <ActivityWorkspace :project-id="projectId" />
  </el-tab-pane>
</el-tabs>
```

`ActivityWorkspace` 在 mounted 阶段立即初始化 ECharts：

```ts
onMounted(async () => {
  await activityStore.syncLogs(props.projectId);
  await nextTick();
  renderHeatmap();
  window.addEventListener("resize", () => chart?.resize());
});
```

`renderHeatmap()` 中只在 chart 不存在时初始化：

```ts
if (!chart) {
  chart = echarts.init(heatmapContainer.value);
}
```

日历横向单元格宽度设置为自动计算：

```ts
cellSize: ["auto", 13]
```

### 2.3 根本原因

Element Plus 非活动 TabPane 通常通过隐藏内容实现标签切换。`ActivityWorkspace` 初始化时“活动”页尚未显示，热力图容器的可用宽度可能是 0、极小值或旧布局值。

ECharts 会在 `echarts.init()` 时读取容器尺寸并创建固定尺寸画布。由于 `cellSize` 的横向尺寸是 `auto`，它会基于这个错误宽度计算 52 周日历格子、月份标签和 visualMap。

用户切换到“活动”标签后：

- 浏览器窗口本身没有发生 resize。
- 现有代码只监听 `window.resize`。
- 没有监听 Tab 激活。
- 没有监听容器尺寸变化。
- 因此 `chart.resize()` 不会执行。

截图中“外层卡片正常但内部图表挤成一团”正是隐藏容器初始化后未 resize 的典型表现。

### 2.4 次要设计问题

1. `window.addEventListener("resize", () => chart?.resize())` 使用匿名函数，组件卸载时无法移除，反复进入项目详情可能累积监听器。
2. 没有在 `onUnmounted` 中调用 `chart.dispose()`。
3. `renderHeatmap()` 每次同步只调用 `setOption`，没有先确认容器已经可见并具有合理尺寸。
4. 当前先构造 `[week, day, count]`，随后又重新转为日期序列，week/day 中间结构没有实际必要，增加了日期边界出错的可能性。
5. tooltip 只显示任务数，没有显示对应日期，用户难以确认所选格子。

### 2.5 推荐修复方案

建议同时采用以下两层措施，而不是只写一个延时器：

#### 方案 A：活动页延迟挂载

给活动 TabPane 增加 `lazy`：

```vue
<el-tab-pane label="活动" name="activity" lazy>
  <ActivityWorkspace :project-id="projectId" />
</el-tab-pane>
```

这样组件首次创建时标签页已经处于可见状态。

#### 方案 B：使用 ResizeObserver

在 `ActivityWorkspace` 内监听热力图容器尺寸：

```ts
let resizeObserver: ResizeObserver | null = null;

onMounted(() => {
  resizeObserver = new ResizeObserver(() => {
    chart?.resize();
  });
  resizeObserver.observe(heatmapContainer.value!);
});

onUnmounted(() => {
  resizeObserver?.disconnect();
  chart?.dispose();
  chart = null;
});
```

ResizeObserver 能同时覆盖：

- 标签页从隐藏到显示。
- 侧栏宽度变化。
- 窗口缩放。
- 页面内容区域因其他组件变化而调整。

#### 方案 C：标签切换后主动 resize

如果不使用 lazy，应在父组件监听 `activeTab`。当值变成 `activity` 时，在两次 `nextTick` 或布局完成后调用子组件暴露的 `resize()`。

不建议只使用固定 `setTimeout(300)`，因为不同电脑、WebView 和动画速度下仍可能复现。

### 2.6 验收标准

- [ ] 初次进入项目详情默认显示“文件”，切到“活动”后热力图横向铺满卡片。
- [ ] 星期标签不重叠。
- [ ] 月份标签按时间顺序分布在热力图顶部。
- [ ] 52 周格子宽度一致，无明显压缩或溢出。
- [ ] 缩放窗口后图表自动适配，不需要刷新页面。
- [ ] 在文件/活动标签间连续切换 10 次，图表布局保持正确。
- [ ] 离开项目详情后 ECharts 实例和 ResizeObserver 被释放。
- [ ] tooltip 同时显示日期和任务数量。

## 3. UI-P2-003：空状态“添加第一个项目”按钮畸变

### 3.1 用户可见现象

截图中的“添加第一个项目”按钮表现为：

- Plus 图标异常巨大。
- 图标与文字不在同一水平线上。
- 按钮高度被异常撑大。
- 图标透明度过低，看起来像一条浅蓝色大十字。

页面右上角的普通“添加项目”按钮显示正常，说明 Element Plus 按钮组件本身没有问题，问题只发生在 `.empty-state` 内。

### 3.2 根本原因

全局样式包含：

```css
.empty-state .el-icon {
  font-size: 48px;
  margin-bottom: 16px;
  opacity: 0.4;
}
```

项目空状态模板同时包含两个 `.el-icon`：

```vue
<div class="empty-state">
  <el-icon><FolderOpened /></el-icon>
  ...
  <el-button type="primary" :icon="Plus">
    添加第一个项目
  </el-button>
</div>
```

Element Plus 会在按钮内部为 `:icon="Plus"` 生成 `.el-icon`。CSS 后代选择器 `.empty-state .el-icon` 不只匹配顶层 FolderOpened 图标，也匹配按钮内部的 Plus 图标。

因此按钮内部图标被错误应用：

- `font-size: 48px`
- `margin-bottom: 16px`
- `opacity: 0.4`

这三条规则共同造成截图中的大按钮畸变。

该问题不仅影响“收藏项目”，还可能影响：

- 全部项目空状态。
- 最近使用空状态。
- 项目回收站空状态按钮。
- 资料回收站空状态按钮。
- 其他在 `.empty-state` 内使用带图标按钮的页面。

### 3.3 推荐修复方案

最稳妥的方案是为展示图标增加专用类，不再依赖后代选择器：

```vue
<el-icon class="empty-state-icon"><FolderOpened /></el-icon>
```

```css
.empty-state-icon {
  font-size: 48px;
  margin-bottom: 16px;
  opacity: 0.4;
}
```

如果希望最小修改，也可以使用直接子元素选择器：

```css
.empty-state > .el-icon {
  font-size: 48px;
  margin-bottom: 16px;
  opacity: 0.4;
}
```

专用类方案更可靠，因为部分页面可能在 `.empty-state` 与图标之间增加包装层。

不建议仅针对按钮覆盖：

```css
.empty-state .el-button .el-icon { ... }
```

这种做法会形成互相覆盖的补丁链，后续更难维护。

### 3.4 验收标准

- [ ] 空状态主图标保持约 48px。
- [ ] 按钮内部 Plus 图标恢复 Element Plus 默认大小。
- [ ] 图标和按钮文字垂直居中。
- [ ] 按钮高度与页面右上角普通按钮一致或符合设计尺寸。
- [ ] 全部项目、收藏、最近、项目回收站和资料回收站空状态逐页检查。
- [ ] 增加计算样式或截图测试，不能只断言按钮文本存在。

## 4. UI-P1-002：服务器资料“按住显示”无效

### 4.1 当前代码行为

“按住显示”按钮目前绑定的是：

```vue
<el-button
  @click="revealContent"
  @blur="maskContent"
  @mouseleave="maskContent"
>
  按住显示
</el-button>
```

这段实现没有监听任何“按下”和“松开”事件。

### 4.2 根本原因一：交互事件与文案不一致

真正的按住显示应当是：

```text
pointerdown → 解密并显示
pointerup / pointercancel / pointerleave → 立即隐藏并清除明文
```

当前逻辑是：

```text
click（完整按下并松开以后）→ 解密并显示
blur / mouseleave → 隐藏
```

所以它从事件定义上就是“点击显示”，不是“按住显示”。

### 4.3 根本原因二：v-if 卸载按钮造成立即回遮罩

解密完成后，store 会设置：

```ts
contentRevealed.value = true;
```

模板随后从遮罩分支：

```vue
<div v-if="!vaultStore.contentRevealed" class="vault-masked">
```

切换到明文分支：

```vue
<div v-else>
  <pre class="vault-content">...</pre>
</div>
```

这会直接卸载原来的“按住显示”按钮。按钮卸载时可能失去焦点，或者鼠标被视为离开原元素，从而触发 `blur` / `mouseleave` 对应的 `maskContent()`。

最终表现可能是：

- 内容闪现后立即消失。
- 解密完成前就重新遮罩。
- 用户感觉按钮完全没有效果。

### 4.4 根本原因三：异步解密存在竞态

`revealContent()` 是异步操作：

```ts
selectedContent.value = await vaultApi.getContent(id);
contentRevealed.value = true;
```

如果用户按下后很快松开，而解密稍后才完成，需要保证“松开状态”优先。当前没有 `isHolding`、请求序号或取消标记，无法处理：

- 指针松开早于解密完成。
- 快速连续按下。
- 触摸 pointercancel。
- 切换资料条目时旧请求稍后返回。

### 4.5 推荐交互模型

“按住显示”和“显示内容”应当是两套明确分离的状态：

#### 临时查看

- 使用 Pointer Events，不要只用 mouse 事件。
- `pointerdown` 开始临时查看。
- 在 `window` 上监听一次性 `pointerup` 和 `pointercancel`，因为显示明文后原按钮可能被卸载。
- 如果解密尚未完成就松开，解密返回后不得再次显示。
- 松开时立即调用 `maskContent()` 清除 Pinia 中的明文。
- 支持键盘 Space/Enter 的 keydown/keyup。

伪代码：

```ts
const isHolding = ref(false);
let holdGeneration = 0;

async function startHold() {
  const generation = ++holdGeneration;
  isHolding.value = true;
  installWindowReleaseListeners();
  await vaultStore.revealContent(selectedId);
  if (!isHolding.value || generation !== holdGeneration) {
    vaultStore.maskContent();
  }
}

function stopHold() {
  isHolding.value = false;
  holdGeneration++;
  vaultStore.maskContent();
  removeWindowReleaseListeners();
}
```

#### 固定显示

- “显示内容”仍然使用 click。
- 解密完成后保持显示。
- 用户可点击“隐藏”。
- 应根据 `vault_auto_mask_seconds` 自动重新遮罩。
- 切换资料、切换路由、窗口失焦或应用隐藏时应清除明文。

### 4.6 相关遗漏：自动遮罩设置没有接入

设置中存在：

```ts
vault_auto_mask_seconds: 30
```

设置页也允许修改“资料显示自动遮罩时间”，但代码库中没有使用该值启动任何遮罩计时器。

当前“显示内容”会一直保持明文，直到用户手动点击“隐藏”、切换资料或其他代码调用 `maskContent()`。这与设置页面表达的行为不一致。

建议：

- 固定显示成功后启动 timer。
- 再次显示时重置 timer。
- 手动隐藏、切换条目、组件卸载时清理 timer。
- timer 到期时调用 store 的 `maskContent()`，确保同时清空 `selectedContent`。

### 4.7 其他相关问题

遮罩线宽度在模板中直接使用：

```vue
:style="{ width: 60 + Math.random() * 30 + '%' }"
```

任何响应式更新都会重新计算随机宽度，遮罩线可能跳动。建议使用固定宽度数组，避免不必要的视觉抖动和非确定性测试。

### 4.8 验收标准

- [ ] 鼠标按下“按住显示”后，保持按下期间显示内容。
- [ ] 鼠标松开后立即隐藏并清除 store 明文。
- [ ] 指针移出、pointercancel、窗口失焦时立即隐藏。
- [ ] 快速点按、长按和连续按压均无闪烁或残留。
- [ ] 解密未完成前松开，解密返回后仍保持遮罩。
- [ ] 触摸屏 pointerdown/pointerup 正常工作。
- [ ] 键盘 Space/Enter 可完成等价临时查看。
- [ ] “显示内容”点击后保持显示，与“按住显示”语义明确区分。
- [ ] 固定显示达到 `vault_auto_mask_seconds` 后自动隐藏。
- [ ] 切换资料或离开资料库页面后 store 中 `selectedContent` 为 null。

## 5. 测试缺口分析

当前项目新增的 `ProjectListView.test.ts` 主要断言：

- “添加项目”文本存在。

- 搜索框和筛选项存在。

1. 文本存在不代表按钮计算样式正确。
2. jsdom 没有真实布局尺寸，不能自然发现 ECharts 在隐藏容器中初始化。

建议新增：

### 5.1 组件测试

- 空状态按钮的图标不应继承 `.empty-state-icon` 大图标样式。
- Vault 临时查看的 pointerdown/pointerup 状态测试。
- 解密 Promise 延迟返回时的竞态测试。
- 使用 fake timers 验证自动遮罩。

- 在真实 WebView 中进入项目详情，切换活动标签，读取热力图 canvas 尺寸。

- 截图检查星期/月标签是否重叠。
- 检查空状态按钮的 bounding box 和图标尺寸。

## 8. 最终分析结论

三个界面问题并非 Windows 缩放、显卡或用户数据导致：

- 热力图问题来自隐藏标签页初始化和缺失容器 resize。
- 大按钮问题来自全局 CSS 后代选择器误伤按钮图标。
- “按住显示”问题来自错误的事件模型、v-if 卸载和异步竞态。

根因均位于前端代码，可稳定修复。修复后必须进行真实 Release 界面复验，因为现有 jsdom 单元测试无法完整覆盖尺寸计算和 Pointer Events 交互。
