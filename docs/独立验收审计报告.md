## 5. 缺陷清单

### AUD-P0-001：业务子路由完全没有渲染

**严重级别：P0 / 阻断发布**  
**状态：确认复现**

路由将 `AppLayout.vue` 配置为父组件，并把项目、收藏、最近、详情、回收站、资料库和设置配置为其子路由：

```ts
{
  path: "/",
  component: () => import("@/layouts/AppLayout.vue"),
  children: [/* 业务页面 */],
}
```

但 `AppLayout.vue` 的内容区域使用普通插槽：

```vue
<main class="page-content">
  <slot />
</main>
```

它没有 `<router-view />`，因此 Vue Router 无处挂载子页面。同时各业务页面本身又各自包裹了一层 `<AppLayout>`，说明当前代码混用了“父布局路由”和“页面自行套布局”两套架构。

**用户影响：**

- 无法添加项目。
- 无法搜索、收藏、重命名、移除或恢复项目。
- 无法进入项目详情、文件树、预览、活动热力图。
- 无法打开服务器资料库和设置页。
- 所有核心功能虽存在代码，但从实际界面均不可使用。

**建议修复：**

采用最小改动方案：把当前路由改为扁平路由，移除父级 `AppLayout` 路由，继续保留每个页面内部已有的 `AppLayout`。修复后增加首页路由冒烟测试，至少断言“添加项目”和“添加第一个项目”可见。

### AUD-P1-002：“检查日志规则”仍为占位代码

**严重级别：P1**  
**状态：确认**

项目列表“检查日志规则”菜单没有调用 `projectApi.checkInstructionFiles`，只读取一次 Pinia 状态后直接提示成功：

```ts
const result = await projectStore.$state; // placeholder
ElMessage.info("日志规则检查完成");
```

这会产生虚假成功提示，无法发现 AGENTS.md 或 CLAUDE.md 管理块缺失、旧版或损坏。

### AUD-P1-003：批量移除功能不可用

**严重级别：P1**  
**状态：确认**

`batchMode` 初始化为 `false`，未发现任何按钮或代码将其设置为 `true`；表格也没有绑定 `selection-change` 来同步选中项目。完成报告中的“批量软移除”后端能力存在，但前端用户流程不可用。

### AUD-P1-004：开机启动设置不产生实际效果

**严重级别：P1**  
**状态：确认**

设置页修改 `launch_at_login` 时只写入 SQLite 和内存设置，没有调用 Tauri autostart 插件的 `enable` 或 `disable`。插件虽然已注册，但设置开关与插件行为没有连接。

### AUD-P1-005：全局快捷键设置不产生实际效果

**严重级别：P1**  
**状态：确认**

程序启动时固定注册 `Ctrl+Alt+P`。设置页修改 `global_shortcut` 只更新数据库，没有注销旧快捷键并注册新快捷键，也没有处理快捷键冲突和失败回滚。

### AUD-P1-006：窗口状态只保存、不恢复

**严重级别：P1**  
**状态：确认**

程序销毁窗口时尝试保存宽度、高度和位置，但启动阶段没有读取这些字段并恢复窗口状态。`window_maximized` 也没有实际更新和恢复逻辑。完成报告中的“窗口状态保存”仅完成了部分链路。

### AUD-P1-007：前端 ESLint 配置失效

**严重级别：P1 / 质量门禁失败**  
**状态：确认复现**

ESLint 对 `.vue` 文件直接使用 `@typescript-eslint/parser`，没有使用 `vue-eslint-parser` 和 `eslint-plugin-vue` 的 flat config。结果是 13 个 Vue 文件均在模板起始位置报 `Parsing error: '>' expected`。

修复后必须确保：

```text
pnpm lint
```

以 0 错误、0 警告退出。

### AUD-P1-008：严格 Clippy 门禁失败

**严重级别：P1 / 质量门禁失败**  
**状态：确认复现**

`cargo clippy -- -D warnings` 产生 27 个错误，而不仅是完成报告中描述的 18 个普通警告。问题包括：

- 未使用 import 和变量。
- 不可达的 `toml` 匹配分支。
- 未使用枚举变体。
- `needless_return`。
- `should_implement_trait`。
- `useless_format`。
- `type_complexity`。
- `manual_strip`。
- `let_unit_value`。
- `useless_vec`。

其中多数是清理类问题，但“不可达模式”和多个未使用变量也说明部分实现没有真正接入。

### AUD-P1-009：自动化测试未覆盖核心用户流程

**严重级别：P1**  
**状态：确认**

27 个前端测试只覆盖：

- `format.test.ts`：16 项。
- `sanitize.test.ts`：11 项。

没有以下测试：

- 路由和布局渲染。
- 项目列表和添加项目对话框。
- 项目详情页。
- 批量选择和移除。
- 设置开关的真实副作用。
- 资料库界面。
- Tauri command mock 集成。

这是 P0 路由错误能够在“全部测试通过”情况下进入安装包的直接原因。

### AUD-P1-010：添加项目失败时存在部分文件副作用

**严重级别：P1**  
**状态：代码审计发现**

添加项目时依次写入 `.project-manager.json`、AGENTS.md、CLAUDE.md 和 `pm_log`。如果 AGENTS.md 已成功写入而 CLAUDE.md 或后续步骤失败，代码只删除标识文件，不恢复 AGENTS.md。本次写入的管理块会残留。该行为不会截断用户原文件，但不符合完成报告中笼统宣称的“失败回滚”。

建议在写入前保存各目标文件的存在状态和原始字节，失败时仅回滚本次操作确实产生的修改，并保留 BOM、CRLF 和原始内容。

### AUD-INFO-011：安装包未签名（本机自用已接受）

**严重级别：信息项，不属于当前缺陷**

安装包 Authenticode 状态为 `NotSigned`。项目所有者已明确当前版本仅在本机自用，因此无需购买或配置代码签名证书，未签名不作为当前验收、安装和使用门槛。只有未来对外分发时才需要重新评估代码签名和可信时间戳。

### AUD-P2-012：品牌图仍带棋盘格背景

### AUD-P2-013：仓库没有 Git 历史

## 6. 安全实现复核

### 6.1 已确认存在的安全实现

以下能力有实际代码和测试支撑，不属于文字占位：

- PathGuard 对已存在路径执行 canonicalize，并检查目标是否位于项目根目录内。
- 相对路径拒绝盘符和明显的 `..` 逃逸。
- 终端通过 `std::process::Command` 和参数数组启动，没有将项目路径拼接成 shell 命令字符串。
- Codex 使用 `resume` / `resume --last`，Claude Code 使用 `-r` / `-c`。
- Markdown 设置 `html: false`，并使用 DOMPurify 二次净化。
- SVG 通过 DOMPurify SVG profile 处理，并禁止 `script` 和 `foreignObject`。
- DPAPI 使用 CurrentUser 范围，并带 `PMV1` 魔数前缀。
- DPAPI 解密失败返回错误，没有发现自动覆盖损坏密文的逻辑。
- Tauri capability 没有开放通用文件系统接口或任意 shell execute 权限。

### 6.2 尚不能宣布完全通过的安全项目

- 现有 PathGuard 测试没有覆盖 Windows Junction、目录符号链接和“父目录是链接、目标文件尚不存在”的组合。
- Markdown/SVG 测试集中于少量固定 payload，没有进行更广泛的协议和编码绕过测试。
- DPAPI 明文会进入前端进程内存以供展示，这是功能必需，但“内存清零”无法覆盖 JavaScript 字符串复制。
- 安装包未签名；本机自用场景已接受该风险，不影响当前验收结论。仍应通过报告记录的 SHA-256 核对安装包是否被意外替换。
- 核心页面当前不可达，因此安全功能尚未经过真实 UI 到 Rust 命令的端到端验证。

安全结论：底层安全设计方向合理，但目前只能认定为“实现存在且单元测试通过”，不能认定为完成端到端安全验收。
